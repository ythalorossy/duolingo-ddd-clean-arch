Design · §4 — Edges, config, scope & acceptance

The last review before I write the spec. Nothing new to decide — confirm this captures it.

⚠ Edges & failure handling

  • Tick fails (DB blip): logged + swallowed; loop survives; retried next interval.
  • Shutdown: stoppingToken ends the loop cleanly. Each week is its own committed unit of work — no half-settled week.
  • No overlap: single awaited PeriodicTimer loop → a long tick coalesces the next, never runs two at once (per instance).
  • Early N+1 activity: Slice-2's reconcile (row created at carried tier, then PlaceInto on settle) still applies — prompt settling just shrinks the window.
  • Multi-instance: two Hosts = two schedulers could collide on the marker insert. Common case safe via the marker; true distributed locking is out of scope.

⚙ Configuration (Leagues:Settlement)

  • Enabled — bool, default true; set false in the 4 test hosts.
  • PollInterval — TimeSpan, default 1h.
  • Added to appsettings.json; bound via LeagueSettlementOptions.

⊘ Out of scope (deferred)

  • Subscriber for Promoted/Demoted (still none).
  • Multi-instance coordination; exact-midnight precision.
  • Divisions/matchmaking, Diamond tournament, demotion grace.
  • Outbox / durable delivery; any change to the settlement algorithm.
  • The live end-to-end smoke test (we chose to skip).

✓ Acceptance criteria

  • An ended week with standings settles automatically within one interval — no manual POST.
  • Catch-up: several ended-but-unsettled weeks settle oldest-first on the next tick.
  • Idempotent: a tick with nothing due moves no one, writes no marker.
  • The current (not-yet-ended) week is never settled.
  • A failing tick logs and does not stop the scheduler; next interval retries.
  • Host shutdown stops the loop cleanly.
  • Existing suite stays green: scheduler disabled in all 4 hosts; manual endpoint & Slice-1/2 behavior unchanged.
  • New Application/Domain types reference nothing infrastructural (NetArchTest).
  • No DB migration introduced.
