Design §1 — the core model (one uniform rule)

The whole feature collapses into a single idea that drives both the write path and the read projection. No special "partial gap" policy needed — it falls out for free.

The rule
gap = whole days missed between lastDate and the date
consumed = min(gap, FreezeBalance)
streak survives ⇔ consumed == gap

One freeze burned per missed day, up to what you hold. Cover the whole gap → streak lives. Run out partway → all remaining freezes burn and the streak still resets. (Faithful to Duolingo, and there's no separate policy to decide.)

then lastDate=D; longest=max(longest,current); raise StreakAdvanced (+ StreakFrozen if consumed>0). Persisted.

READ — Report(today)  ·  pure, no mutation
today vs lastStatus
today = lastActive (current)
gap=0 (last=yesterday)AtRisk (current)
gap≥1, balance ≥ gapAtRisk; freezes=balance−gap
gap≥1, balance < gapBroken (0)

Same gap math as the write — so what you see before studying is exactly what gets persisted when you do.

Why this stays "derived, no nightly job"

The stored FreezeBalance is only decremented by a real qualifying activity. A read never mutates — it projects "balance − consumed" against today. Miss Thu with 2 freezes: Friday's read shows AtRisk, 1 freeze left (Thu projected as covered) without writing anything; Friday's lesson then persists exactly that (balance 2→1, current +1). If you never come back, each read just recomputes against an ever-larger gap until it shows Broken. Stored state stays put; the projection does the work.

Cap = 2 freezes (a domain constant, easily tuned). Look right? Reply in the terminal — next I'll cover components, then errors/edge-cases, then tests.
