Slice 2 — how settlement runs

An explicit SettleLeagueWeek(week) command (the seam) settles every tier's cohort for that week, idempotently. NEW = built in Slice 2.

The settlement flow
POST /leagues/weeks/{week}/settle (NEW) → SettleLeagueWeek(week) (NEW)
↳ already settled? (week marker exists) → no-op (idempotent)
↳ otherwise, for each tier Bronze…Diamond:
rank the tier's week-N cohort → k = floor(0.2 × N) → top k ↑ (Diamond stays) bottom k ↓ (Bronze stays) → write each mover's week N+1 placement
then insert week-settled marker (NEW)
Example — Gold cohort of 12 settles into next week

↑ top 2 → Sapphire

8 stay → Gold

↓ bottom 2 → Silver

Movers get a fresh week-N+1 row (new tier, 0 XP). Stayers' next row carries their tier forward when they next play.

Idempotency & the "no auto-trigger yet"

A per-week settled marker makes re-running SettleLeagueWeek a no-op — same "idempotency is a domain rule" theme as the AppliedAward ledger.

What automatically closes a week (a scheduler, or lazy-on-activity) is deferred — Slice 2 builds the operation; a later increment wires the real driver. The command/endpoint is the seam (like GrantStreakFreeze).
