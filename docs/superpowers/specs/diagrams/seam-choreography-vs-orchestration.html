The Learning → Engagement seam: how should they talk?

When a learner finishes a lesson, Engagement must award XP, advance the streak, and update league points. How does Engagement find out?

Option A — Choreography (publish an event)
Flow: Learning grades the attempt and raises LessonCompleted. It does not know who cares. Engagement subscribes and reacts. So do Notifications, future Achievements, etc.

Learning ──LessonCompleted──▶ (bus/outbox) ──▶ Engagement
                           └─▶ Notifications

Coupling: Learning depends on NOBODY. The supporting context stays ignorant of the core. ✅
Option B — Orchestration (a use-case coordinates)
Flow: A SubmitLessonAttempt application service calls Learning.Grade(), then explicitly calls Engagement.AwardXp(), then Notifications — all in one place.

SubmitLessonAttempt ─▶ Learning.Grade()
                  ├─▶ Engagement.AwardXp()
                  └─▶ Notifications.Send()

Coupling: the orchestrator knows about ALL contexts. Add a new reactor → edit the orchestrator. ⚠️

A — Choreography / events

  • Core (Engagement) evolves independently; new reactors just subscribe — no edits upstream
  • Learning has zero knowledge of XP/streaks — clean supporting→core direction
  • Naturally extensible (Achievements, Notifications, Analytics all listen)
  • Maps directly to future service extraction (the evolutionary goal)

A — costs

  • Eventual consistency: XP may land milliseconds later — UI must handle it
  • Harder to trace ("why no XP?") — needs good logging/observability
  • Needs an outbox to publish reliably (no lost events)

B — Orchestration / direct calls

  • Dead simple to read and debug — one linear use-case
  • Can be one atomic transaction (strong consistency)
  • Great when there's exactly one consumer and it'll stay that way

B — costs

  • Centralizes knowledge of every context in the orchestrator
  • Every new reactor (achievements, analytics) edits this hot path
  • Couples the core's API surface to the lesson-submit flow

Which seam do you want?

A

Choreography via events (my recommendation)

In-process domain event now → integration event via outbox as we evolve.

B

Orchestration via direct calls

Simpler, atomic, but the use-case knows every context.

C

Hybrid

Orchestrate the must-be-atomic bits; publish events for the optional reactors.
