Q3 · The streak model — states, transitions, and a key insight

A streak is driven by one event ("a qualifying day happened, local date D") plus the passage of time.

Transition function — when a qualifying day occurs on local date D (given stored count, lastDate)

ConditionEffect
D == lastDateno-op — already counted today (idempotent)
D == lastDate + 1count += 1; lastDate = D  (streak continues)
D > lastDate + 1 (gap)count = 1; lastDate = D  (reset & restart)
D < lastDateignore (late/out-of-order event)

💡 The key insight: streaks are DERIVED, not scheduled

Lazy / derived (recommended)

  • Store only count + lastQualifyingDate
  • "At risk" / "broken" are computed on read vs today's local date — not stored
  • A missed day needs no action: next time we observe the streak, the gap reveals it's broken
  • No nightly job, no per-user timers — scales trivially

Eager / scheduled (rejected)

  • A nightly batch resets everyone's broken streaks
  • Needs scheduling infra, scans all users, timezone-staggered runs
  • Classic over-engineering for state you can just derive
