How should the background service decide when & what to settle?

A .NET BackgroundService runs a loop for the lifetime of the Host. The design choice is what that loop does each time it wakes. Two strategies — the difference shows up the moment the Host isn't running exactly at midnight Monday.

Scenario that separates them: the Host was down across two week boundaries, then comes back during week 4
Weeks of league activity · ✓ settled · ✗ ended but never settled · ▦ current
W1 ✓
W2 ✗
W3 ✗
W4 ▦ now
⟵ Host offline during the W2→W3 and W3→W4 boundaries ⟶   (deploy gap, crash, weekend downtime…)
A

Periodic poll + idempotent catch-up  ▰ recommended

Wake on a fixed interval (say hourly). Each tick: "find every ended week with no settlement marker, settle them oldest-first." On the W4 restart it settles W2 then W3 in order — the chain stays correct (W2's promotions create W3's rows before W3 is ranked). Robust to any downtime; self-bounding (only weeks with data); reuses the Slice-2 idempotency marker as the "have I done this?" memory. Fires within one interval of the boundary, not exactly at midnight.

tick → settle W2 → settle W3 → W4 stays open (not ended) → done
B

Sleep until the next boundary

Compute the next Monday 00:00 UTC, delay until exactly then, settle the one week that just ended, repeat. Precise — fires right at the boundary. But it slept through the W2 and W3 boundaries while the Host was down, so those are lost… unless you also add catch-up-on-startup — at which point you've rebuilt strategy A anyway, plus fragile deadline math (clock changes, long delays).

restart → next wake = W4→W5 boundary → W2, W3 never settled

Why A is the better teacher, too: "a loop that polls + an idempotent operation that owns its own done-marker" is the exact backbone of real schedulers, outbox processors, and reconciliation jobs. You'll also hit the canonical .NET lesson head-on: a BackgroundService is a singleton, so it can't take the scoped mediator/DbContext directly — it opens an IServiceScopeFactory scope per tick. And the interval timer can use the injected TimeProvider, so FakeTimeProvider drives the whole thing deterministically in tests — no real waiting.
