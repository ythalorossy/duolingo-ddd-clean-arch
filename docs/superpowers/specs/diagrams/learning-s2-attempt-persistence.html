Q6 — Does the Attempt get persisted?

This is the scope hinge of Slice 2. In Slice 1, Learning was read-only at runtime (load a lesson, publish an event, write nothing). An Attempt that persists gives Learning its first write path — new table, repository, migration, SaveChanges. The alternative keeps grading transient and Learning read-only.

B2 — Persist the Attempt (real write path) recommended

POST /lessons/{id}/attempts { answers[] } → load Lesson (+ exercises) → lesson.Grade(answers) → GradingResult → new Attempt(learner, lessonId, answers, result) → SaveChanges ← persists EVERY attempt (pass or fail) → if Outcome == Passed: publish LessonCompleted → XP → 200 { score, outcome, per-exercise correctness }

Why

A never-persisted "aggregate" is a contradiction — aggregates are consistency/persistence boundaries. This is the honest first write path, and the durable Attempt history is exactly what Slice 3 (once-per-lesson credit, reduced XP on repeat) reads.

Cost

One learning.Attempts table + AttemptRepository + migration + first SaveChanges. Modest — a skeleton growing its spine, not gold-plating.

B1 — Transient grading (Learning stays read-only)

POST /lessons/{id}/attempts { answers[] } → load Lesson · lesson.Grade(answers) → GradingResult → if Passed: publish LessonCompleted → XP (nothing written) → 200 { score, outcome, correctness }

Why

Maximally restrained — preserves Slice 1's clean read-only property; no table/migration. Defers all attempt storage to Slice 3.

Cost

The "Attempt aggregate" from Q4 becomes just a GradingResult value object — we walk back the aggregate. Slice 3 then has to build the write path and backfill history it never captured. Repeats/pass-rate are unobservable until then.

Event timing (both options): publish only on a pass, and in B2 after SaveChanges — the persisted Attempt is the source of truth; the in-process XP award stays best-effort (same no-outbox posture as Slice 1; a future outbox could re-publish from the saved row). No Learning domain-event dispatcher — LessonCompleted is an integration event the handler publishes directly, exactly as today. Still repeatable: each passing attempt awards XP again; true "once per lesson" is Slice 3.
