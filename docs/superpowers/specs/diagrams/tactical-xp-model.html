Phase 4 · Tactical model — Engagement XP slice

One aggregate, strongly-typed value objects, one domain event. Small on purpose — but modeled properly.

Aggregate: LearnerEngagement (root)
Identity: LearnerId (value object wrapping the Identity UserId — Conformist)

State: TotalXp : Xp

Behavior:
AwardXp(XpAward award)
  • enforces invariants
  • increases TotalXp
  • raises XpAwarded

Invariants (live in the aggregate):
  • award amount > 0
  • TotalXp never negative / never overflows
  • one award per LessonCompleted (idempotent)
Value Objects & Event
Xp — wraps a non-negative int. Xp + XpAward → Xp. No primitive int XP floating around (kills primitive obsession).

XpAward — amount + reason (e.g. LessonCompleted) + source id for idempotency.

LearnerId — strongly-typed id, not a raw Guid.

Domain event: XpAwarded
(LearnerId, amount, newTotal, occurredOn)
raised inside the aggregate, dispatched in-process within the same transaction.

The request flow (this is the walking skeleton)

1. POST /lessons/{id}/complete → maps to CompleteLesson command (current user = faked for now)
2. Learning stub handles it and raises LessonCompleted(LearnerId, LessonId)
3. Engagement subscribes → loads/creates LearnerEngagement → AwardXp(policy.For(lesson))
4. Aggregate raises XpAwarded; repo persists to SQL Server engagement schema (1 transaction)
5. GET /me/engagement → query handler → returns { totalXp }

One design decision I made for you (challenge it)

XP policy lives in ENGAGEMENT, not Learning

  • Learning only announces "a lesson was completed." It doesn't know XP exists.
  • Engagement owns "how much is it worth" — that's a core-domain rule (and will get rich later: combos, boosts, weekend XP).
  • For the skeleton: a trivial flat policy (e.g. 10 XP). The seam is what matters, not the number.

The alternative (rejected)

  • Learning emits the XP amount in the event.
  • Rejected: it leaks an Engagement concern into a supporting context and couples them.
