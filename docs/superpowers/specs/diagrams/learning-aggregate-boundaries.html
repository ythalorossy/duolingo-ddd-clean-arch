Where do the aggregate boundaries fall?

An aggregate is a transactional consistency boundary — and the unit you load & save together. The tree is Course → Unit → Lesson; the question is how many boxes we draw around it.

The tell A thing referenced from outside by a global id is an aggregate root. Here, lessons are already addressed by global id — POST /lessons/{lessonId}/complete and LessonCompleted.LessonId. That fingerprint matters below.
A

One aggregate — Course owns Units & Lessons

The whole tree is one consistency boundary. Units and Lessons are entities inside Course, with only local identity.

🔒 Aggregate root: Course
📘 Course “Spanish”
📂 Unit “Basics”
📄 Lesson “Greetings”
📄 Lesson “ser”

+ Ordering invariants (unit/lesson sequence) enforced in one place. − To complete lesson X you must find & load its whole Course — but the endpoint only has a lessonId. Fights how we actually address lessons.

B

Three aggregates — Course, Unit, Lesson, linked by id recommended

Each is its own root. A Unit holds a CourseId; a Lesson holds a UnitId. No object holds another aggregate — only its id.

🔒 Course
📘 Course
◀ holds CourseId
🔒 Unit
📂 Unit
◀ holds UnitId
🔒 Lesson
📄 Lesson

+ Completion loads just the Lesson by its id — exactly what Slice 1 needs. Small aggregates, references by identity (the house style). − Cross-tree invariants (ordering) aren't transactional — fine for seeded reference data.

C

Two aggregates — Course owns Units; Lesson stands alone

The small, cohesive course/unit structure is one aggregate; the globally-referenced, growing Lesson is its own root holding a UnitId.

🔒 Course (root) + Units
📘 Course
📂 Unit “Basics”
📂 Unit “Food”
◀ holds UnitId
🔒 Lesson
📄 Lesson

+ Structure stays cohesive; the hot, referenceable entity is independent. − Two boundary styles to hold in your head — a touch more than a skeleton needs.
