What is the aggregate?

The leaderboard is "everyone in my tier, this week, ranked by weekly XP." Two very different ways to model that — and the difference shows up as a write-contention trap.

A · Per-learner LeagueStanding  ★ recommended
LeagueStanding (aggregate · one per learner)
  • Id : LearnerId
  • Tier : LeagueTier (Bronze…Diamond — all Bronze in Slice 1)
  • Week : LeagueWeek (the UTC Monday)
  • WeeklyXp : Xp
RecordXp(amount, whenUtc) → lazy-rolls the week (reset if Week is stale) then adds. Same lazy-settle trick as LearnerStreak.
Ana · 980 Bruno · 910 Chen · 845 …each its own small row
Leaderboard = read-model query
standings WHERE Tier=mine AND Week=now ORDER BY WeeklyXp DESC
CQRS read, like GetLearnerStreak. No cross-aggregate JOIN — one filtered read of the standings table.

Wins: small, uncontended writes · mirrors LearnerStreak exactly · idempotent & simple.

Slice-2 consequence: settlement is a cross-aggregate process (rank the query, update each standing) — which is the cohort-wide-settlement problem we already earmarked for Slice 2.

B · LeagueCohort aggregate, key (Tier, Week)
LeagueCohort (aggregate · one per Tier+Week)
  • Tier, Week
  • Members : [ {LearnerId, WeeklyXp}, … ] (all of them)
RecordXp(learner, amount) · Settle() ranks members & emits promote/demote.
Ana 980 · Bruno 910 · Chen 845 · Dilan 720 · Eva 690 · … all in one object

Win: Slice-2 settlement is a clean single-aggregate method cohort.Settle().

Trap: every learner's XP gain mutates this one shared, ever-growing object → write contention + a large-aggregate smell. Real Duolingo caps divisions at ~30 precisely to bound this; our Plan-B cohort (a whole tier) is unbounded.

Which aggregate model for Slice 1?

A

Per-learner LeagueStanding + read-model leaderboard ★

Small uncontended aggregates; mirrors LearnerStreak; settlement becomes a Slice-2 process.

B

LeagueCohort aggregate (Tier, Week)

Settlement-as-method is elegant, but a hot, unbounded, contended aggregate on the write path.
