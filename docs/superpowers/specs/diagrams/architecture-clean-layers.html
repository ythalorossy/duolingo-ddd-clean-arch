2b · Clean Architecture inside a module

The Dependency Rule: source-code dependencies point INWARD only. The domain knows nothing about EF Core, HTTP, or any other module.

interfaces (depends on NOTHING) dependencies point inward Dependency Inversion Domain/App declare interfaces (IXpRepository). Infrastructure writes the implementation (EF Core). Composition root wires them via DI at startup.

Where each piece lives (the placement that trips people up)

LayerWhat lives hereDepends on
DomainAggregates, Entities, Value Objects, Domain Events, Domain Services, Repository interfaces, invariants/business rulesnothing
ApplicationUse cases (commands/queries + handlers), DTOs, port interfaces (e.g. IEmailSender), unit-of-work orchestration, integration-event publishingDomain
InfrastructureEF Core DbContext + mappings + migrations, repository implementations, outbox, adapters to email/push/paymentApplication + Domain
PresentationAPI endpoints, request/response models, auth wiring, mapping to commands/queriesApplication

How do we structure the Application layer's use cases?

A

CQRS-lite with a mediator (recommended for learning)

Each use case is a Command/Query message + Handler. Pipeline behaviors handle validation, logging, transactions cross-cuttingly. Very idiomatic .NET DDD.

B

Plain application services

Classes with methods (XpService.AwardXp(...)). Less ceremony, less "magic", but cross-cutting concerns are wired by hand.
