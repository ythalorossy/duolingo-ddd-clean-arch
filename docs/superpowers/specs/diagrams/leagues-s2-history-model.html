Settlement needs week N's standings — but Slice 1 overwrites them

To rank a cohort, settlement must read every member's final week-N XP. Slice 1 keeps one mutable row per learner and overwrites it on the next week's first activity. Here's the fix.

A · One row per learner (Slice 1, today)
LeagueStanding key = LearnerId
Ana · Gold · Week N · 980
▼ Ana earns 15 XP in week N+1 → RecordXp rolls & overwrites
Ana · Gold · Week N+1 · 15

Broken: the 980 (week N) is gone. Settlement of week N has nothing to rank.

B · One row per (learner, week)  ★ recommended
LeagueStanding key = (LearnerId, Week)
Ana · Gold · Week N · 980  (kept) Ana · Gold · Week N+1 · 15  (new row)

Works: week N persists as history; N+1 accumulates in its own row. Settlement reads week-N rows whenever it runs.

What changes with the recommended model (B)

  • RecordXp find-or-creates the row for the current week instead of overwriting — old weeks are never touched.
  • The leaderboard query is already keyed by (tier, week), so it fits with no change.
  • Settlement reads week-N rows, then writes each learner's week-(N+1) placement (their new tier).
  • Edge: a learner who earns in N+1 before settlement runs gets their tier carried forward, and settlement reconciles it (their N+1 XP follows them to the right tier). Acceptable under the explicit-settlement (C) model.
  • Cost: a migration changing the primary key from LearnerId → (LearnerId, WeekStart), and rows accumulate over weeks (fine; prunable later).
B

Per-(learner, week) rows ★

Evolve LeagueStanding's identity to (LearnerId, Week). History preserved; settlement reads it directly.

A

Keep one row + a separate week-result snapshot

Write an immutable snapshot at week close. Rejected: the snapshot must be written before the overwrite — timing-fragile, and it duplicates the standing.
