Design · §1 — Architecture & data flow

The scheduler is just another driving adapter — a second way to pull the same trigger the HTTP endpoint already pulls. Nothing about the Slice-2 settlement operation changes. We add a timer that drives a thin new policy command, which reuses the existing per-week operation.

Each tick (and the catch-up case from before)
timer fires → open DI scope → resolve IMediator → Send(SettleDueLeagueWeeks)
  ↳ due = [W2, W3] (ended, have standings; W4 is current → skipped)
  ↳ Send(SettleLeagueWeek W2) → commits W2 promotions and the W2 marker
  ↳ Send(SettleLeagueWeek W3) → now sees W2's movers in W3's cohorts → commits
  ↳ next tick: W2/W3 markers exist → both no-op. Nothing to do until W4 ends.

Three deliberate calls baked in here: (1) the policy ("what's due") lives in Application, so it's testable with zero hosting; the BackgroundService is pure mechanism (timer + scope + lifecycle). (2) We reuse SettleLeagueWeek rather than re-implement — and sending it once per week gives each week its own committed transaction, which the settlement chain requires. (3) Catch-up needs no new memory — the Slice-2 marker already records "done," so the loop is naturally idempotent.
