Where should freeze state live?

Semantics are settled. The real design decision is the aggregate boundary — and the DDD rule that decides it: an aggregate is a consistency boundary. Things that must change together, atomically, belong together.

A — Freeze count on LearnerStreak  ★ recommended
LearnerStreak <<aggregate>>
CurrentStreak · LongestStreak
LastQualifyingDate
FreezeBalance (new)
─────────────
RegisterQualifyingActivity()
GrantFreeze() · Report(today)

Bridging a gap = decrement balance and keep the streak — one object, one transaction. Consumption is persisted on the next qualifying activity; Report projects it purely on read.

B — Separate StreakFreezeInventory aggregate
LearnerStreak
streak state
⇄
FreezeInventory
balance, items

Auto-apply now spans two aggregates: either one transaction touching both (couples them anyway) or eventual consistency via events (gap → FreezeApplied → streak update). Heavier; the consistency story gets hard.

Also considered — and rejecting

C — A nightly settlement job (per-timezone midnight: burn a freeze or break the streak). It makes "freeze applied" a real timed event, but reintroduces exactly the per-user timers / scheduled work the streaks design deliberately avoided ("derived on read, no nightly job"). Freeze should extend that same lazy model, not abandon it.

Which boundary do you want?

A

One aggregate — FreezeBalance on LearnerStreak ★

Streak + freeze change atomically together. Richest single invariant, no background job, stays lazy. My recommendation.

B

Two aggregates — separate inventory

More extensible (typed items, expiry) but adds cross-aggregate coordination we don't need yet (YAGNI).

C

Nightly settlement job

Timed freeze application. Abandons the no-job lazy model the streak design committed to.
