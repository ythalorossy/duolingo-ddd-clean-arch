2a · Macro Architecture — the Modular Monolith

One deployable (the API host). Modules are isolated as if they were services. Angular is just one client of the API.

The rules that make it a MODULAR monolith (not a big ball of mud)

  • No direct cross-module references. Learning cannot new an Engagement class. Modules see each other only through a thin Contracts assembly (events + DTOs) and the event bus.
  • Each module is internally layered (Clean Architecture — that's screen 2b). Its domain model is internal; nothing leaks out.
  • Data isolation is what makes future extraction realistic — no cross-module JOINs, no shared tables. That's the open question below.

Data isolation strategy?

A

Schema-per-module, one physical DB (recommended)

Logical isolation, no cross-schema JOINs allowed. Cheap to run, honest boundaries, easy to split out later.

B

Database-per-module now

Strongest isolation, closest to microservices. More infra/ops overhead for a learning monolith.

C

Shared schema, separated by convention

Simplest, but boundaries erode fast (someone WILL write a cross-module JOIN). Fights the evolutionary goal.
