Q2b · How does Engagement get the learner's time zone?

Time zone is owned by Identity (conceptually). Engagement needs it locally to compute the day boundary. Three ways to bridge that boundary:

Option 1 — Engagement keeps a local replica (recommended)
Identity (truth) ──LearnerTimeZoneChanged event──▶ Engagement stores its own copy
                                                   (default UTC; set via command now,
                                                    Identity event later)
LessonCompleted ─▶ streak uses the LOCAL copy (no call-out)
Each context owns the slice of data it needs, in its own model, updated by events.
Option 2 — time zone on each API request
Client sends X-Time-Zone header with the lesson-complete call.

Simplest to wire, but: no source of truth, breaks for events not driven by that request, and scatters a profile concern across callers.
Option 3 — synchronous call to Identity
Engagement calls Identity ("what's this learner's TZ?") every time it computes a day.

Always fresh, but couples the contexts at runtime, adds latency, and fights the "extractable later" goal.

Why Option 1 (local replica)

  • Engagement stays self-sufficient — no runtime coupling to Identity
  • Teaches that data duplication across contexts is normal (truth + local cache via events)
  • For this slice: set via a small command (Identity is still faked); swap to an event later — no streak logic changes
  • Time zone rarely changes, so a replica is cheap and almost always fresh

The trade-off you accept

  • Eventual consistency: if a learner changes TZ, Engagement's copy updates when the event arrives (fine — TZ changes are rare)
  • A bit more plumbing now (a TimeZoneId field + a set-timezone command)

Decision

1

Local replica on Engagement (recommended)

Store TimeZoneId; default UTC; set via command now, Identity event later.

2

Per-request header

Pass time zone with each call. Simple, but no source of truth.

3

Synchronous call to Identity

Always fresh, but runtime coupling + latency.
