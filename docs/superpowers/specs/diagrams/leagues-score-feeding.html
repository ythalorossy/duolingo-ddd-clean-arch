How does the weekly score get fed?

A learner earns XP → their league weekly score must go up. There are three ways to wire that, and they differ in where the XP amount comes from and how coupled leagues are to XP. Boxes = handlers/aggregates; arrows = flow.

A · React to LessonCompleted, re-apply the XP policy
LessonCompleted → AwardXp handler → XpAccount  & in parallel  → ⚠ Leagues handler recomputes XP via policy → LeagueStanding +XP

Trap: the XP amount is computed in two places. Add a bonus-XP rule later and the league total silently drifts from the real XP.

B · React to XpAwarded via a domain-event dispatcher  ★ recommended
LessonCompleted → AwardXp → XpAccount → raises XpAwarded(amount) → Dispatcher (NEW) → Leagues handler +amount → LeagueStanding

Wins: counts the actual XP (single source of truth) · leagues don't know why XP was earned (future-proof) · builds the dispatcher the codebase already flagged as "arrives when a subscriber exists" · idempotent for free — XpAwarded is raised at most once per award (the AppliedAward ledger guarantees it), so event re-delivery can't double-count.

Cost: a small new building block; dispatch is in-process best-effort (no outbox yet → same maturity as today).

C · One handler awards XP and bumps the score, atomically
LessonCompleted → one handler: AwardXp + bump LeagueStanding → save XpAccount + LeagueStanding (1 txn)

Trade: atomic and simple, knows the exact amount — but one handler now has two responsibilities and writes two aggregates in one transaction (against the "one aggregate per transaction" guideline), and it skips the event-driven decoupling the project favors.

Which wiring for Slice 1?

A

React to LessonCompleted, recompute XP

Simplest wiring, no new infra — but duplicates the XP calculation.

B

React to XpAwarded via a new dispatcher ★

Correct, decoupled, idempotent for free, builds the deferred building block. My recommendation.

C

One atomic handler does both

Atomic and simple, but couples two responsibilities + two aggregates per transaction.
