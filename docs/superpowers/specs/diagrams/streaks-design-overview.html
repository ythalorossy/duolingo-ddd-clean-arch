Streaks — design overview

Top-down data flow. Two aggregates in the Engagement context now both react to LessonCompleted.

1 · A lesson is completed
POST /lessons/{id}/complete → Learning.Stub raises LessonCompleted → published on the in-process bus.
↓   fan-out to two independent subscribers   ↓
Subscriber 1 (existing)
AwardXpForLessonCompleted
→ XpAccount
renamed from LearnerEngagement
Subscriber 2 (new) ⭐
RegisterStreakForLessonCompleted
→ LearnerStreak
TimeProvider + LearnerTimeZone → local date → transition
⭐ LearnerStreak — the new aggregate (keyed by LearnerId)
State: TimeZoneId (LearnerTimeZone value object) · CurrentStreak · LongestStreak · LastQualifyingDate (DateOnly?, null until first activity)

Behavior:
• RegisterQualifyingActivity(utcInstant) — convert to local date, run the 4-case transition, bump longest
• ChangeTimeZone(tz) — update the local replica
• GetStatus(today) — derived read → Active / AtRisk / Broken + effective current

idempotent per local DAY — no AppliedAward-style ledger needed
Set time zone
PUT /me/timezone → SetLearnerTimeZone command → ChangeTimeZone
Read streak
GET /me/streak → GetLearnerStreak query → GetStatus(today) (derived)
Persistence & time
SQL table engagement.LearnerStreaks (DateOnly → date column). TimeProvider injected as the clock so tests can pin "today" and walk a learner across days.
