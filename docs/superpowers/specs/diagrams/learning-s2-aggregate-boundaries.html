Q4 — Where do exercises live?

Slice 2 introduces exercises (multiple-choice) and an Attempt. The key DDD call: is an Exercise an entity inside the Lesson aggregate, or its own aggregate root? The deciding lens is the same one Slice 1 used: addressability — plus a new one: where the correct answer is allowed to exist (it must never leave the server).

A — Exercises live inside the Lesson aggregate recommended

Lesson becomes a real aggregate root that owns a collection of Exercise entities. The correct answer 🔒 lives on the Exercise, inside the Lesson boundary — it never crosses into a DTO. Attempt is its own root, referencing the lesson by id.

Lesson aggregate (1 aggregate → 2 tables)
Lesson (root) · Id · UnitId · Title · IsPublished · PassThreshold
Exercise (entity) · prompt · choices · correctChoice 🔒
Exercise (entity) · prompt · choices · correctChoice 🔒
Attempt aggregate
Attempt (root) · Id · LearnerId · LessonId (by id) · answers[] · Score · Outcome

✅ Mirrors Engagement's XpAccount + owned AppliedAwards — the codebase's proven "1 aggregate / 2 tables" shape (Slice 1 never had children). Grading loads one Lesson with its exercises, grades in one boundary. The answer key has a natural home that a read model simply can't reach.

B — Exercise is its own aggregate root (by id)

Consistent with Slice 1's "small roots referenced by id" (Course/Unit/Lesson). Grading loads the lesson's exercises via a query WHERE LessonId = ….

Three independent roots
Lesson (root) · Id · UnitId · Title · IsPublished
Exercise (root) · Id · LessonId (by id) · prompt · choices · correctChoice 🔒
Attempt (root) · Id · LearnerId · LessonId · answers[] · Score · Outcome

⚠️ Exercises aren't addressed from outside by a global id — nobody does GET /exercises/{id}; they're always reached through their lesson. That's the tell of a child entity, not a root. Making it a root also scatters the answer key across a standalone table and invites "load exercises separately" races the aggregate would prevent.

Either way: Attempt is its own aggregate root (it has its own lifecycle and global id), and the correct answer never appears in any read model — the "present the exercise" DTO (Slice 2 also needs a GET to show prompts + choices) omits it. Grading is server-authoritative.
